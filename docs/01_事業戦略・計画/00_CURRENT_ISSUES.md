# 現状の問題点整理

**作成日:** 2026-03-22

---

## 1. 事業戦略書の問題点

### 1.1 事業定義が「送客型メディア」に留まっている
- エグゼクティブサマリーが「工務店とマッチングする」「マーケティング支援サービスを提供する」で終わっている
- 消費者の意思決定プロセスを支援する、という視点がない
- 住宅会社への提供価値が「YouTubeの集客力で見込み客を獲得」＝単なるリード送客になっている
- データを蓄積して何に使うか、という将来構想が欠落

### 1.2 競争優位性が表面的
- 「YouTube登録者42,841人」「動画257本」という定量面だけ
- 平屋特化という市場ポジショニングの深掘りがない
- データ蓄積による参入障壁の構築という観点がない
- 競合比較が「SUUMO / HOME'S / YouTube住宅系」の3つだけで浅い（※v2.0で修正済み。なお、唯一の平屋特化ポータルであったe平屋net（Lib Work社）は2026年1月31日に閉鎖。現在、日本に平屋専門のプラットフォームは1つも存在しない。ぺいほーむが日本初かつ唯一となる）

### 1.3 収益モデルにデータ活用・AI活用の将来収益がない
- 全10項目がすべて既存の「制作代行」「掲載料」「仲介」型
- AIコンシェルジュ利用料、意向データ分析、営業支援SaaSなどの将来収益がゼロ
- ストック収入の比率が低く、スケールしにくい構造

### 1.4 ロードマップがフェーズ名だけで進化ストーリーがない
- Phase 0〜3の記載はあるが「何が何につながるか」が不明
- 「データ蓄積→AI進化→SaaS化」という段階的な進化の接続がない
- AIチャットボットがPhase 3で突然出てくるが、そこに至るデータ基盤の話がない

### 1.5 データ戦略・AI戦略の章が存在しない
- テクノロジー戦略はあるが「技術スタック一覧」で終わっている
- どのデータを取り、何に使い、どうAIを進化させるかが完全に欠落

## 2. 要件定義書の問題点

### 2.1 機能・画面の列挙に偏っている
- 62画面の一覧は充実しているが「何のデータを取得するか」の視点がない
- 各画面に「取得すべきイベント」「蓄積すべきデータ」が紐付いていない
- 物件詳細ページの要件に「閲覧をuser_eventsに記録する」などの記載がない

### 2.2 データモデルがトランザクション処理に限定
- users / leads / builders / events の4テーブルのみ
- ユーザーの行動履歴（user_events）が存在しない
- チャットセッション・メッセージの保存構造がない
- ユーザープロフィール（家族構成・予算・検討フェーズ等）の概念がない
- お気に入り・比較履歴のテーブルがない
- コンテンツエンゲージメントの集計テーブルがない

### 2.3 leadの定義が問い合わせフォームの送信データに過ぎない
- 「流入チャネル」「直前閲覧履歴」「チャット要約」「関心タグ」「不安要素」がない
- 推薦理由・検討フェーズ・温度感が欠如
- リードが「フォーム送信の瞬間」しか捉えておらず、そこに至る行動文脈が失われている

### 2.4 AIチャットの要件が1項目で雑
- 「OpenAI GPT-4o / SSE / 1,000トークン」という実装仕様だけ
- FAQ応答・ヒアリング・推薦・診断・要約・リード化判定などの機能分解がない
- MVP / 次フェーズ / 将来の段階分けがない
- 会話ログの保存・活用について未記載

### 2.5 ユーザーダッシュボードが📋（後回し）になっている
- お気に入り・比較・閲覧履歴がPhase 2以降
- これらはデータ蓄積の中核であり、AIコンシェルジュの学習データ源
- 後回しにすると、データが溜まらないまま時間が過ぎる

### 2.6 CMS要件がコンテンツ入稿の箱に留まっている
- 工務店・物件・記事の登録項目にAI推薦用の構造化データ（デザインテイスト・性能・向いている顧客像等）がない
- CMSで登録されたデータがAIの推薦材料になる、という接続が設計されていない

### 2.7 イベント計測・分析要件が存在しない
- GA4連携は🔧だが、カスタムイベントの定義がない
- どのページでどのイベントを発火し、何を計測するかが未定義
- KPIがプラットフォーム全体の粗いものだけで、ファネル別・AI別のKPIがない

### 2.8 開発優先順位がデータ基盤より機能拡張寄り
- Phase 1が「メール通知」「CMS永続化」「KPIレポート」
- ユーザー行動取得・チャットログ保存・イベント計測が先であるべき
- 「見た目の機能は動くが、データが溜まらない」状態が長引くリスク

## 3. AI進化観点で不足している要素

| 不足要素 | 影響 |
|---------|------|
| ユーザー識別子の統一設計 | 匿名→会員のデータ接続ができない |
| 行動イベントの定義・取得 | AIが学習するデータが存在しない |
| チャットログの構造化保存 | 会話から意向抽出・要約ができない |
| ユーザープロフィールの段階的構築 | パーソナライズの基盤がない |
| コンテンツの構造化タグ | AI推薦のマッチング材料がない |
| リードと行動データの紐付け | 「なぜこの人が問い合わせたか」が分からない |
| 比較・お気に入りの保存 | ユーザーの検討状況が可視化できない |
| 意向推定ロジックの設計 | 温度感・検討フェーズの自動判定ができない |

## 4. 優先順位のズレ

### 現状の優先順位（問題あり）
```
Phase 0: コンテンツ公開 + CMS + リード獲得
Phase 1: メール通知 + CMS永続化 + KPIレポート  ← 機能寄り
Phase 2: RBAC + ユーザーDB + Stripe             ← データ蓄積が遅い
Phase 3: AIチャット高度化                         ← データなしで高度化できない
```

### あるべき優先順位
```
Phase 0: コンテンツ公開 + CMS + リード獲得（✅ 完了）
Phase 1: ユーザー識別 + 行動取得 + チャットログ保存 + リード拡張 ← データ基盤
Phase 2: お気に入り/比較/履歴 + CMS構造化 + AI推薦初期版 ← データ活用開始
Phase 3: AIコンシェルジュ高度化 + 工務店向けレポート + SaaS化 ← AI本格稼働
```

**核心的な問題：** 現状の文書は「動く機能を作る」ことに最適化されており、「将来のAI活用のためにデータを溜める」という視点が設計レベルで入っていない。
