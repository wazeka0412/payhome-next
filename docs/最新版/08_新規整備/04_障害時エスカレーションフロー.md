# 障害時エスカレーションフロー

最終更新: 2026-04-12
対象: 全スタッフ (特に初動対応する営業・CS・PM)

---

## 1. 障害レベルの定義

| レベル | 症状 | 影響 | 初動 SLA |
| --- | --- | --- | --- |
| **P0** | サイトが全く動かない / DB 破損 / データ漏洩 | 全ユーザー | **5 分以内に代表+技術** |
| **P1** | 主要フローが動かない (会員登録・見学会予約・AI診断) | 多数ユーザー | **15 分以内に技術リード** |
| **P2** | 一部機能が動かない / 特定ユーザーだけ発生 / 軽微なデータ不整合 | 一部ユーザー | **1 時間以内に技術** |
| **P3** | 表示崩れ / 文言ミス / 軽微な UI バグ | 限定 | **翌営業日** |

---

## 2. 初動対応 (誰が最初に検知した場合でも共通)

### ステップ 1: 検知 (30 秒)

- エラー画面のスクリーンショットを撮る
- 発生時刻・URL・ユーザー環境 (ブラウザ / デバイス) をメモ
- Sentry / Vercel Logs / Supabase Dashboard を確認

### ステップ 2: レベル判定 (30 秒)

- 自分で動作再現できるか?
- 複数ユーザーが影響を受けるか?
- 主要フロー (会員登録・見学会予約・AI診断) が止まっているか?
- P0 or P1 と判定したら即座に次のステップ

### ステップ 3: Slack エスカレーション (1 分以内)

Slack `#payhome-incident` チャンネルに以下テンプレートで投稿:

```
【P0 or P1】障害検知
- 発生時刻: HH:MM
- 症状: (簡潔に)
- 再現手順: 1. ... 2. ...
- 影響範囲: (全ユーザー / 特定フロー / 特定ユーザー)
- 初動担当: (自分の名前)
- スクリーンショット: (添付)
- Sentry URL: (あれば)
```

### ステップ 4: 該当担当者を @メンション

- **P0**: `@代表 @技術リード @PM` 全員
- **P1**: `@技術リード @PM`
- **P2**: `@技術リード`
- **P3**: チケット化のみ

---

## 3. 担当別の対応フロー

### 営業・CS が検知した場合

1. 上記ステップ 1〜4 を実行
2. 影響を受けたユーザーがいれば、**暫定対応として謝罪メッセージ**:
   ```
   「ご迷惑をおかけして申し訳ございません。
    現在サイトで障害が発生しており、復旧作業を進めています。
    復旧次第、改めてご連絡いたします。」
   ```
3. 技術側の復旧を待つ
4. 復旧後、該当ユーザーに完了連絡

### PM が検知した場合

1. 上記ステップ 1〜4 を実行
2. 技術リードにエスカレーション
3. 代表と相談してステークホルダー連絡の要否を判断
4. 15 分以上経過しても解決しない場合、状況アップデートを Slack に投下

### 技術リードが検知した場合

1. 即座に原因調査 (Sentry / Logs / DB)
2. 同時に Slack でエスカレーション
3. 修正/ロールバックを決定
4. 復旧後、Post-mortem を起票

## 4. 典型的な障害と対応

### 4.1 全ページ 500 エラー

**原因候補**: 環境変数ミス / DB 接続失敗 / Vercel 側障害

**対応**:
1. Vercel ダッシュボードで直近のデプロイを確認
2. 環境変数を確認
3. Supabase ダッシュボードでサービス状態確認
4. 問題解決しなければロールバック (Vercel で Promote)

### 4.2 会員登録で /welcome が 404

**原因候補**: middleware の HIDDEN_PATH_PREFIXES に /welcome が入っている (履歴バグ)

**対応**:
1. `src/middleware.ts` を確認
2. `/welcome` がコメントアウトされていることを確認
3. 変更があれば即コミット → デプロイ

### 4.3 AI診断が動かない

**原因候補**: OpenAI API クォータ超過 / API キー失効

**対応**:
1. `https://platform.openai.com/usage` でクォータ確認
2. `OPENAI_API_KEY` の有効性確認
3. 必要なら新規 API キー発行して再デプロイ

### 4.4 Supabase クエリエラー多発

**原因候補**: RLS ポリシー誤設定 / スキーマ変更失敗

**対応**:
1. Supabase → SQL Editor で該当クエリ実行
2. RLS ポリシー確認
3. マイグレーションロールバック検討

### 4.5 見学会予約通知メールが届かない

**原因候補**: Resend API キー失効 / 送信制限

**対応**:
1. Resend ダッシュボードでログ確認
2. API キー再発行
3. 緊急時は手動で工務店に電話連絡

### 4.6 決済エラー (Phase 4 以降)

**原因候補**: Stripe Webhook 失敗

**対応**:
1. Stripe ダッシュボード → Webhooks でログ確認
2. 手動で Webhook 再送
3. 関係する顧客に連絡

## 5. 障害連絡テンプレート

### 社内通知

```
【INCIDENT】ぺいほーむ 障害発生
レベル: P0 / P1 / P2 / P3
発生時刻: YYYY-MM-DD HH:MM
症状: (簡潔に)
影響範囲: (全ユーザー / 特定フロー / 件数)
初動担当: (氏名)
次の報告: HH:MM
```

### 復旧通知

```
【RESOLVED】ぺいほーむ 障害復旧
レベル: P0 / P1 / P2
発生時刻: HH:MM / 復旧時刻: HH:MM
影響時間: XX 分
原因: (簡潔に)
対応: (簡潔に)
影響ユーザー: XX 名
Post-mortem: (予定日 YYYY-MM-DD)
```

### 工務店向け通知 (P0/P1 で 30 分以上の場合のみ)

```
Subject: 【お詫び】ぺいほーむ サイト障害のお知らせ

平素よりぺいほーむをご利用いただき、誠にありがとうございます。

本日 HH:MM 頃より、ぺいほーむのサイトで以下の障害が発生しておりました:

発生時間: HH:MM 〜 HH:MM (約 XX 分間)
影響: (簡潔に)
現状: 復旧済み / 復旧作業中

ご迷惑をおかけし、誠に申し訳ございません。
原因の詳細と再発防止策は別途ご報告いたします。

株式会社wazeka ぺいほーむ事業部
```

## 6. Post-mortem テンプレート

障害発生後 3 営業日以内に作成:

```markdown
# Post-mortem: YYYY-MM-DD [障害タイトル]

## 概要
- レベル: P0 / P1 / P2
- 発生時刻: YYYY-MM-DD HH:MM
- 復旧時刻: YYYY-MM-DD HH:MM
- 影響時間: XX 分
- 影響ユーザー数: XX 名
- 影響ビジネス指標: (見学会予約 XX 件遅延 等)

## タイムライン
- HH:MM: ○○が△△を検知
- HH:MM: 技術リードがエスカレーション受領
- HH:MM: 原因特定
- HH:MM: 修正デプロイ
- HH:MM: 復旧確認

## 原因
(技術的な根本原因を簡潔に)

## 影響
- ユーザー側: (何が起きたか)
- 社内側: (何が起きたか)
- ビジネス: (金額換算できれば)

## 対応
- 即時対応: (何をしたか)
- 恒久対策: (何をする予定か)

## 再発防止策
1. (コード修正 / テスト追加 / 監視強化 / 手順変更 等)
2. ...

## 学び
- 良かったこと (検知が早かった 等)
- 改善すべきこと (対応が遅れた 等)

## Action Items
- [ ] (担当者) (期日) (タスク)
- [ ] ...
```
